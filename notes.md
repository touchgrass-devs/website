## Performance pass (2026-07-16)
User reported the page feeling laggy. Audited every canvas/WebGL/animation-loop component; two fixes shipped so far (rest of the punch list - Threads always-on shader, navbar `backdrop-blur` on scroll, unoptimized 1.9MB hero source jpeg, `filter: blur(140px)` ambient orbs, 42-image external-CDN logo marquee - not yet actioned):
- **`PhilosophyDotGrid.jsx` - the big one.** Its canvas is `absolute inset-0` inside the Philosophy section, which (once the `h-[340vh]` pinned-dial track is in normal flow) makes the canvas several thousand px tall - thousands of dots + the constellation line mesh were being redrawn on *every animation frame, forever*, gated only by `document.hidden` (not visibility), so the loop ran even while scrolled nowhere near Philosophy. Fixed two ways: (1) added an `IntersectionObserver` on the wrapper that fully stops/starts the `requestAnimationFrame` loop based on whether any part of the section is on screen, matching the pattern `Threads.jsx` already used; (2) `draw()` now computes the visible row band each frame (via the wrapper's `getBoundingClientRect()` plus cached row pitch/start-Y from `buildGrid`, stored in new `pitchRef`/`startYRef`) and only touches dots/lines inside that band (+150px buffer), instead of iterating the whole multi-thousand-dot canvas regardless of what's actually on screen. Dot-draw and line-draw are still two separate full passes over that band (not interleaved) so line color never reads a one-frame-stale neighbor `_t` - same ordering the original relied on, just scoped smaller. Zero visual change; same dots draw the same way whenever they're actually in view.
- **`Services.jsx` re-render storm.** The coverflow's drag rotation (`liveOffset`) and per-card hover-glow (`mousePos`) both lived as `useState` on the top-level `Services` component, so every drag frame *and* every mousemove over any one of the 4 cards re-rendered the entire section - all four cards, the header, the nav buttons, the caption - not just the thing that changed. Split each card into its own `ServiceCard` component: hover-glow is now local `useMotionValue`s + `useMotionTemplate` (zero React re-renders, ever - Framer patches the DOM style directly), and drag-offset is now a local per-card subscription to the shared `spinSteps` motion value (still triggers a re-render, since Framer's `animate` prop is what gives button-click selection its smooth spring-to-target transition and needs a real prop value - but now scoped to just that one card's subtree, not the whole section). No behavior change intended; same spring config, same button-click smoothing, same hover glow look.
- Both verified via `@babel/parser` (jsx plugin) against the exact written content - this session's bash sandbox has a long, unpredictable sync lag against the mounted project folder (confirmed via `stat`: mtimes lag ~1h+ behind real edits), so `next dev`/`eslint` run from this sandbox were reading stale/truncated file snapshots and not usable for verification this round. Worth a real `npm run build` locally before treating this as fully verified, same caveat already logged elsewhere in this file for other changes.
- **Round 2 (same day): removed the double smooth-scroll and the blur-filter ambient orbs.**
  - **CSS `scroll-behavior: smooth` (globals.css) removed entirely.** It was running alongside Lenis's own scroll-driven rAF loop (`SmoothScroll.jsx`) - both were trying to own "smooth scrolling" for the whole page at once, which can fight (native smooth-scroll animating the real scrollTop while Lenis is simultaneously re-asserting its own virtual position each frame). Lenis is now the single source of smooth scroll site-wide. This CSS rule was also the *only* thing making plain anchor-link clicks (nav links, mobile drawer, Hero's CTA/scroll-down button) animate smoothly at all - none of them went through Lenis before. Rather than let those regress to instant jumps, added a small `scrollToHash(hash)` click-handler (in both `Navbar.jsx` and `Hero.jsx` - small enough, and different enough in each file's specifics, not worth a shared `lib/` module per the project's own no-premature-abstraction rule) that calls `window.__lenis.scrollTo(target, { offset: -96, duration: ... })` when Lenis is mounted, falling back to native `scrollIntoView({ behavior: 'smooth' })` otherwise (covers the brief pre-mount window and the explicit `prefers-reduced-motion` case, since `SmoothScroll.jsx` doesn't instantiate Lenis at all under reduced motion). `offset: -96` mirrors the `scroll-mt-24` every target section already carries, so links still land below the fixed nav exactly as before. Wired onto: the brand mark (`#hero`), all 4 desktop nav items, the desktop `MagneticCTA` (now takes an `onNavigate` prop), all 4 mobile drawer items and the mobile CTA (combined with their existing drawer-close `onClick`), and Hero's own `#contact` CTA and "Scroll to Explore" button (previously a bare `scrollIntoView` call, now routed through the same helper for consistency). Philosophy's dial-node `goToIndex` already did this exact `window.__lenis` pattern beforehand - untouched, already correct.
  - **`.premium-blur-orb` (globals.css) reworked - no more `filter: blur(140px)`.** That's one of the most expensive CSS paint operations, and being static (all 6 usages - 2 each in Hero/Expertise/Contact - are plain, non-animating `<div>`s) doesn't make it free: anything else repainting in the same stacking context can force a static blurred element to re-rasterize too. Replaced with a `mask-image: radial-gradient(circle, black 0%, black 15%, transparent 100%)` on the same element - a soft, borderless falloff via compositing instead of a blur convolution, same look at near-zero cost. Also dropped the class's `will-change: transform` - it existed only to force these onto their own GPU layer to cache the blur's output; per `design-taste-frontend`'s own rule ("use will-change sparingly, only on elements that actually animate"), it shouldn't have been on static shapes regardless. Zero JSX changes needed in Hero/Expertise/Contact - the existing `bg-{color}/5 premium-blur-orb` classNames are untouched, only the CSS behind `.premium-blur-orb` changed. **One caveat, unlike the scroll fix above:** a mask can't bleed color past the element's own box the way `filter: blur()` does, so the glow is now strictly contained within each div's existing (generously-sized) bounds rather than softly extending a bit beyond it - should read as effectively identical given how subtle these already are (5% opacity, corner-positioned, several partially off-screen already), but worth an actual visual glance locally since this couldn't be screenshotted in this sandbox.
  - Verified the same way as the round above: reconstructed the edited files' exact content in a scratch path and parsed with `@babel/parser` (the mounted path itself was showing a stale/truncated snapshot again at verification time - same sandbox sync-lag issue, not a real bug). Not yet visually checked locally.

- **Untried next steps if this gets picked up again:** the package's own `@splinetool/react-spline/next` entry point (an async Server Component) is Spline's documented fix for this exact Next.js resolution problem, but using it requires restructuring so the `<Spline>` element is created in a Server Component and passed down through `Contact.jsx` (a Client Component) via composition/children, rather than imported directly inside a `'use client'` file - not attempted here.
